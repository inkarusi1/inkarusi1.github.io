一、操作系统的内核及其启动

为了启动操作系统需要将内核冲虚在计算机上运行起来。内核需要直接与硬件交互来管理各个硬件。

CPU 不能直接访问磁盘；内存 RAM 是易失性寄存器。CPU 能直接访问的非易失性寄存器：ROM（不可写） 和 FLASH（空间限制）。但它们的区域大小优先；操作系统内核在 CPU 加电后直接启动使一个计算机只能启动一个操作系统；硬件相关代码全放在操作系统中不利于操作系统的移植工作。

因此将硬件初始化的相关工作（指令不多）作为**”bootloader**”程序放在非易失存储器中，将操作系统内核放在磁盘中。

将硬件启动与软件启动分离。bootloader 为硬件初始化，然后将内核镜像从磁盘读到 RAM 中；操作系统内核则专注于软件启动和对用户提供服务的工作。

二、MIPS 汇编与 C 语言

编译与链接

通过不同阶段的反汇编后文本，我们应该从中理解函数调用时跳转的目的地址变化。

栈帧：栈帧保存了函数的返回地址和局部变量。函数递归操作过程中，把所有变量和状态压入栈中，返回时再弹出来。每一个栈帧对应一个未运行完的函数。栈帧的作用包括但不限于：存储函数的返回地址、存储调用方的临时变量与 中间结果、向被调用方传递参数。

LEAF, NESTED (嵌套的) 和 END

这些可留到后面理解。

我们会在 init/start.S 里看到 LEAF(_start) 这样的代码。这些宏定义以 . 开头，不是 MIOS 汇编指令，主要用于编译是指定目标代码的生成。

在 include/asm/asm.h 中 LEAF, NESTED, END 定义如下：

当使用 LCR 指令执行调用操作时，返回地址将保存在 RPC 寄存器中，而 RPC 中的旧值将保存在堆栈中（以16位方式分两次操作） ，22 位的函数入口地址将被装载到 PC 计数器， 从而使流程转入函数体中运行。当使用 LRETR 指令执行返回操作时，将从 RPC 寄存器中读取返回地址装载到 PC 计数器中，并将堆栈上的值写入 RPC 寄存器中（以16位方式分两次操作）。其他调用指令不使用RPC寄存器。

对于 Makefile 部分语法的解释：

这里:=其实是阻塞赋值

  1. “=”

    make会将整个makefile展开后，再决定变量的值。也就是说，变量的值将会是整个makefile中最后被指定的值。看例子：

    x = foo y = $(x) bar x = xyz

    在上例中，y的值将会是 xyz bar ，而不是 foo bar 。

  2. “:=”

    “:=”表示变量的值决定于它在makefile中的位置，而不是整个makefile展开后的最终值。

    x := foo y := $(x) bar x := xyz

    在上例中，y的值将会是 foo bar ，而不是 xyz bar 了。

  • include ：目录中存放系统头文件。在本章中需要用到的头文件是 mmu.h 文件，这个文件中 有一张内存布局图，我们在填写 linker script 的时候需要根据这个图来设置相应节的加载 地址。
  • init ：内核中各模块的初始化部分都会在这里被调用。目录中主要有两个代码文件 start.S 和 init.c，其作用是初始化内核。start.S 文件中的 _start 函数是 CPU 控制权被转交给内核后执行的第一个函数，主要工作是初始化 CPU 和栈指针，为之后的内核初始化做准备，最后跳转到 init.c 文件中定义的 mips_init 函数

ELF 文件结构

发现了一个特别好的博客：ELF文件格式-第一讲，详解程序文件头 - Android_IBinary - 博客园 (cnblogs.com)

ELF 是一种用于可执行文件、目标文件和库的文件格式。通过 file <file> 可以获取文件类型。

  • 可重定位 (relocatable) 文件，
  • 可执行 (executable) 文件，
  • 共享对象 (shared object) 文件，后两种文件都需要链接器对可重定位文件进行处理才能生成。

ELF 文件的5个部分：

  • ELF 头。以 typedef struct 结构体的形式存储程序（ELF文件）的基本信息：比如体系结构和操作系统，同时也包含了节头表和段头表 相对文件的偏移量（offset）等。假设 binary 为 ELF 的文件头地址，shoff 为节头表入口偏移，那么 binary + shoff 即为节头表第一项的地址。

  • 段头表/程序头表（program header table）。其信息在程序运行时刻使用。包含程序各段（segment）信息。

  • 节头表（section header table）。其信息在程序编译和链接时使用。包含个节（section）信息。

  • 段头表各表项。记录了该段数据载入内存时的目标位置等

  • 节头表各表项。有很多节头，记录了该节程序的代码段、数据段等各个段的内容

    重要的有这3个：

    .text 保存可执行文件的操作指令。

    .data 保存已初始化的全局变量和静态变量。

    .bss 保存未初始化的全局变量和静态变量。

可以视为一个节头表存了很多节头结构体？每个结构体存储对应的节部分的信息。

MMU (Memory Management Unit) 完成地址转换。

  • kuseg (2 GB) ：用户，cache，MMU中TLB变换

  • kseg0 (512 MB) ：内核下直接映射，cache。存储内核的 .text, .data, .bss 段。include/mmu.h 中的 KERNBASE (0x8001 0000) 是内核镜像的起始虚拟地址。

一个程序想要运行：

  1. 如果是裸机程序，则需要对应体系结构的 CPU。运行时把它放到内存的特定位置（这是 bootloader 的活），让 CPU 找到他去运行，我们的 MOS 内核就是这样被运行的。
  2. 如果不是裸机程序，则需要对应体系结构的 CPU，需要有 OS、用户库的支撑，OS 帮你去找到他然后去运行（运行过程可能依赖用户库）。

四、实战 printk

需要掌握的C语言知识

函数参数列表：末尾有省略号时，表明该函数有变长的参数表。传入函数的可能有多个参数表，每个参数表都要有固定的起始位置，因此只能让最后一个参数表长度不定。

  1. va_list，变长参数表的变量类型；保有 va_start 、 va_arg 、 va_end 及 va_copy 所需信息 (typedef)
  2. va_start(va_list ap, lastarg)，用于初始化变长参数表的宏；令函数得以访问可变参数；lastarg 为该函数最后一个命名的形式参数，具体是什么可以看看示例，因为我也说不清
  3. va_arg(va_list ap, 类型)，用于取变长参数表下一个参数的宏；在初始化后，每次可以使用 va_arg 宏 按获取一个形式参数，该宏也会同时修改 ap 使得下次被调用将返回当前获取参数的下一个参数
  4. va_end(va_list ap)，结束使用变长参数表的宏。

一个重要的知识点就是C和C++中泛型编程（比如可以用于适应不同类型参数的函数）。比较常见的泛型编程有C++种的template方法，而这里着重讲一下C语言。

C语言中泛型编程要用到 void *（指向任意类型的指针，之后在函数中再转化类型）。而且在函数中，由于无法对void *直接解引用（?），函数还需要额外传入 size 参数标示出 void * 指向的数据类型的大小。

一些注意点：

  • 变量不能声明为 void 无类型；
  • 不能将 * 用在无类型指针上，因为系统没有此地址指向对象大小的信息（编译阶段，编译器无法得知该传入参数类型）。这也是为什么我们需要在调用的地方传入要交换的对象的地址空间大小 size；
